# TripleA + GPT 全自动对战：项目实现与部署说明

本文档说明：**项目如何从头到尾实现**、**如何写接口与调用 GPT**、**当前支持/不支持的能力**、**如何在别的电脑上复现部署**。

---

## 一、整体架构（三块怎么配合）

```
┌─────────────────┐      LAN (TCP)       ┌─────────────────┐      HTTP (localhost:8081)      ┌─────────────────┐
│  TripleA Host   │ ◄──────────────────► │  Game Bridge     │ ◄─────────────────────────────► │  GPT Driver     │
│  (game-headed)  │   连接 + 接管日本     │  (game-bridge)   │   GET /state, /legal_actions   │  (Python 脚本)   │
│  GUI 建房间     │                      │  Java, 端口 8081 │   POST /act                    │  chatgpt_driver │
└─────────────────┘                      └─────────────────┘                                  └─────────────────┘
         │                                        │                                                      │
         │ 游戏引擎 (GameData, Delegate)          │ 实现 Player 接口，阻塞在 start(stepName)            │ OpenAI API
         │ 购买/部署/移动/战斗 等阶段              │ 等 HTTP 发来动作后调用 Remote 执行                  │ (gpt-4o-mini)
         ▼                                        ▼                                                      ▼
    人类或 AI 对手                           只接管「日本」这一家                                   用 Function Calling
                                                                                                   调用 3 个 tool
```

- **Host**：TripleA 主程序（game-headed），建房间、选地图、跑游戏逻辑。日本可以由「人类」或「网络客户端」控制。
- **Bridge**：以**网络客户端**身份连上 Host，**接管日本**，对外提供 **HTTP API**（state / legal_actions / act）。内部实现 TripleA 的 `Player` 接口，在购买/部署等阶段阻塞，等收到 HTTP 的 act 再通过 RPC 调 Host 的 Delegate 执行。
- **GPT Driver**：Python 脚本，循环「获取 state + legal_actions → 调用 OpenAI Chat Completions + Function Calling → 根据返回调用 POST /act」，实现全自动下日本这一家。

## 二、Bridge 是怎么实现的（接口、API、与引擎对接）

### 2.1 入口与连接

- **BridgeMain**：`main()` 里解析 `--host/--port/--name/--take`，创建 **BridgeRuntime**，在后台线程里 `runtime.connect()`，然后启动 **Javalin** HTTP 服务（端口 **8081**）。
- **connect()**：用 TripleA 的 `ClientMessengerFactory` 连 Host，注册 `IClientChannel`、`IObserverWaitingToJoin`，调用 `serverStartup.takePlayer(localNode, takePlayerName)` 接管指定玩家（如 Japanese）。
- **startGame()**：收到 Host 的 `doneSelectingPlayers(gameData, players)` 后，反序列化 GameData，创建 **ClientGame**，把 **BridgePlayer** 作为「日本」的 Player 实现注入，并调用 `gameLoader.startGame(...)`，从而进入引擎的回合循环。

### 2.2 引擎侧：Player 与 Delegate

- 每个**阶段**（购买、战斗移动、战斗、非战斗移动、部署等）对应一个 **Delegate**（如 PurchaseDelegate、MoveDelegate、BattleDelegate、PlaceDelegate）。
- Host 在**服务器**上跑这些 Delegate；轮到日本时，会通过 RPC 调用**客户端**（Bridge）上的 **BridgePlayer** 的接口。
- **BridgePlayer** 实现 TripleA 的 **Player** 接口：
  - **start(stepName)**：在购买/部署等阶段会**阻塞**，直到 Driver 通过 HTTP 发来 BUY_UNITS / PLACE_UNITS / END_TURN 等，内部调 `signalStepDone()` 才会返回。
  - **performPurchase(...)** / **performPlace(...)**：通过 `PlayerBridge.getRemoteDelegate()` 拿到 Host 上的 **IPurchaseDelegate** / **IAbstractPlaceDelegate**，远程调用 `purchase(...)` / `placeUnits(...)`，把结果返回给 Host。
  - **selectCasualties(...)**：战斗伤亡选择阶段不能返回 null，当前实现为**采用引擎给的默认伤亡**（`new CasualtyDetails(defaultCasualties, false)`），避免 Host NPE。
  - 其他步骤（如移动、政治等）暂未实现具体逻辑，仅通过 **END_TURN** 跳过。

### 2.3 HTTP API（如何写接口）

Bridge 对外就是三个接口（在 **BridgeMain** 里用 Javalin 绑定）：

| 方法 | 路径 | 作用 |
|------|------|------|
| GET | /health | 健康检查，返回 `ok` |
| GET | /state | 返回当前对局的 JSON state（game、japan、territories、unitsByTerritory、purchaseOptions、placeOptions 等），供 GPT 读局面 |
| GET | /legal_actions | 返回当前阶段允许的动作类型列表，如 `[{ "type": "BUY_UNITS" }, { "type": "END_TURN" }]` |
| POST | /act | 执行一个动作，body 为 JSON，如 `{ "type": "END_TURN" }` 或 `{ "type": "BUY_UNITS", "items": [...] }` |

- **getState()**：在 **BridgeRuntime** 里读 `launchAction.getGame().getData()`，在 **BridgeStateBuilder** 里拼出 `game`（round、stepName、phaseName、currentPlayerName、controlledPlayerName）、`japan`（pus、techTokens）、`territories`、`unitsByTerritory`、`purchaseOptions`、`placeOptions`（部署阶段会 RPC 调 placeDelegate.getPlaceableUnits）。
- **getLegalActions()**：根据当前 `GameStep` 判断是购买/部署/战斗移动/非战斗移动/战斗等，往列表里加 BUY_UNITS、PLACE_UNITS、COMBAT_MOVE、NON_COMBAT_MOVE、BATTLE、END_TURN 等（见下一节「当前支持与不支持」）。
- **applyAction()**：根据 `type` 分发到 `applyEndTurn()`、`applyBuyUnits()`、`applyPlaceUnits()`；COMBAT_MOVE / NON_COMBAT_MOVE / BATTLE 目前统一当作 END_TURN 处理（跳过该阶段）。

要**新增一种动作**（例如将来的「移动」）：在 **BridgeRuntime** 里给 `getLegalActions()` 加一种 type，在 `applyAction()` 里加分支，在 **BridgePlayer** 里实现对应步骤（如拿到 **IMoveDelegate** 后调用 `performMove(moveDescription)`），并在 **BridgeStateBuilder** 里如有需要增加 state 字段（例如可移动单位、可移动目标）。

## 三、如何调用 GPT（Driver、tools、prompt）

### 3.1 脚本入口与循环

- **chatgpt_driver.py**：解析 `--base-url`、`--auto`、`--rules-file`，创建 **TripleABridgeClient**（默认 `http://localhost:8081`），组装 **system prompt**（含规则文件内容），然后：
  - 单次模式：执行一轮 `run_one_round()` 结束。
  - 全自动模式：`while True` 里先做**上下文裁剪**（按「用户消息」边界保留最近 2 回合），再 `run_one_round()`，再 `get_state()` 判断是否我方回合/是否断开，然后 append 用户消息「继续」，进入下一轮。

### 3.2 如何写「接口」给 GPT：Function Calling

GPT 不直接「写代码」，而是通过 **OpenAI Function Calling** 暴露三个 **tools**，每个对应一个「接口」：

1. **get_game_state**：无参数。Python 里 `run_tool(client, "get_game_state", {})` → `client.get_state()` → 请求 `GET /state`，把返回的 JSON 作为 tool 结果塞回 messages。
2. **get_legal_actions**：无参数。`client.get_legal_actions()` → `GET /legal_actions`，返回列表如 `[{ "type": "BUY_UNITS" }, { "type": "END_TURN" }]`。
3. **do_action**：参数 `action_type`（必填），以及 `units`（BUY_UNITS）、`placements`（PLACE_UNITS）等。Python 根据 `action_type` 调 `act_end_turn()` / `act_buy()` / `act_place()`，内部是 `POST /act`。

在代码里就是 **TOOLS** 列表：每个元素是 `type: "function"` + `function.name` + `function.description` + `function.parameters`（JSON Schema）。这样模型会按需返回 `tool_calls`，Driver 解析后执行对应 HTTP 调用，再把结果以 `role: "tool"` 消息追加，继续请求 API，直到模型不再发起 tool 调用。

### 3.3 Prompt 与规则

- **SYSTEM_PROMPT**：说明「你控制日本」「只能用三个 tool」「state 里 game.controlledPlayerName 表示我方」「currentPlayerName === controlledPlayerName 才是我方回合」「END_TURN 只结束当前阶段」以及购买→战斗移动→战斗→非战斗移动→部署的阶段顺序。
- **rules-file**：`--rules-file rules_zh.txt` 会把文件内容拼进 system prompt，用于教 GPT 地图规则、日本目标、运输/海战等（当前引擎侧尚未开放移动/战斗动作，所以 GPT 在这些阶段只能 END_TURN 跳过）。

## 四、当前支持与不支持（为什么只会买和部署）

### 4.1 已实现

- **购买阶段**：legal_actions 含 BUY_UNITS、END_TURN；POST /act 的 BUY_UNITS 会调 Host 的 PurchaseDelegate，日本可以买单位。
- **部署阶段**：legal_actions 含 PLACE_UNITS、END_TURN；POST /act 的 PLACE_UNITS 会调 Host 的 PlaceDelegate，日本可以部署新单位。
- **其他阶段（战斗移动、非战斗移动、战斗）**：legal_actions 会包含 COMBAT_MOVE、NON_COMBAT_MOVE、BATTLE（带 note 说明当前仅支持 END_TURN）以及 END_TURN。POST /act 若 type 为 COMBAT_MOVE / NON_COMBAT_MOVE / BATTLE，Bridge 会当作 END_TURN 执行，即**跳过该阶段**，不会进攻、不会移动、不会用运输船、不会海战。
- **战斗伤亡**：若因其他玩家进攻导致日本进入伤亡选择，Bridge 已实现 **selectCasualties** 返回默认伤亡，Host 不会 NPE。

### 4.2 未实现（要扩展才能进攻/移动/运输/海战）

- **战斗移动（Combat Move）**：需要 Bridge 在 Move 阶段拿到 **IMoveDelegate**，向 GPT 暴露「可移动单位 + 目标」或「可执行的 MoveDescription 列表」，并实现 `performMove(MoveDescription)` 的 HTTP 映射（例如 POST /act 的 type: "COMBAT_MOVE" + moves: [...]）。引擎侧已有 `IMoveDelegate.performMove(MoveDescription)`，需要定义 MoveDescription 的 JSON 格式并在 Bridge 里拼装、序列化/反序列化。
- **非战斗移动（Non-Combat Move）**：同上，用同一 MoveDelegate 在非战斗阶段执行移动，包括**用运输船运兵**（load/unload + 移动），都需要在 state 里暴露「可装载单位、可卸载领土」等，并在 act 里传对应 move。
- **海战**：本质是「战斗移动时把单位移入敌占海域/领土触发战斗」+ 战斗阶段的伤亡选择（已支持）。因此实现战斗移动后，海战会随移动一起出现。

若要实现「进攻、移动、运输、海战」，需要：

1. 在 **BridgeRuntime** / **BridgeStateBuilder** 里，在战斗移动/非战斗移动阶段通过 **IMoveDelegate**（及可能的辅助接口）获取可执行的移动/装载/卸载信息，并加入 **GET /state** 的 JSON。
2. 在 **GET /legal_actions** 中保留 COMBAT_MOVE、NON_COMBAT_MOVE，并可能细化为 MOVE、LOAD、UNLOAD 等。
3. 在 **POST /act** 中新增参数（如 moves、loads、unloads），在 **BridgePlayer** 或 Bridge 的 delegate 封装里把 JSON 转成 **MoveDescription**（或引擎需要的结构），调用 `performMove(...)` 或等效接口。
4. 在 **chatgpt_driver** 的 TOOLS 里为 do_action 增加这些参数，并在 system prompt / rules 里教 GPT 如何根据 state 填 moves（包括运输船用法）。

## 五、如何复现部署

### 5.1 环境要求

- **本机**  
  - **Java**：JDK 17+（TripleA 与 Bridge 编译运行需要）。  
  - **Python**：3.10+（推荐 3.11/3.12），用于运行 GPT Driver。  
  - **TripleA 工程**：从仓库 clone 或拷贝整个项目（含 `game-app/game-bridge`、`game-app/game-headed`、`game-app/game-core`、`clients/python` 等）。

- **OpenAI**  
  - 有 **OpenAI API Key**（用于 gpt-4o-mini），并设置环境变量 `OPENAI_API_KEY`。

### 5.2 步骤（按顺序）

1. **编译 TripleA（可选，若已编译可跳过）**  
   ```bash
   cd /path/to/triplea
   ./gradlew build
   ```
   或至少编译 bridge：  
   `./gradlew :game-app:game-bridge:compileJava`

2. **启动 Host**  
   - 运行 TripleA GUI（game-headed），例如：  
     `./gradlew :game-app:game-headed:run`  
   - 建游戏：选地图（如 Pacific 40）、玩家数，**日本**设为「人类」或可由网络客户端接管，端口如 **3300**，不设密码，进入等待玩家界面，**先不要点开始**。

3. **启动 Bridge**  
   新开一个终端：  
   ```bash
   cd /path/to/triplea
   ./gradlew :game-app:game-bridge:run --args="--host 127.0.0.1 --port 3300 --name Bot_Bridge --take Japanese"
   ```  
   - `--host` / `--port` 要和 Host 一致；`--take` 必须是该地图里日本的实际名字（如 `Japanese` 或 `Japan`）。  
   - 看到类似 “Bridge connected and took player: Japanese” 和 “Bridge HTTP server listening on port 8081” 后，在 Host 里点**开始游戏**。

4. **启动 GPT Driver**  
   再开一个终端：  
   ```bash
   cd /path/to/triplea/clients/python
   export OPENAI_API_KEY=sk-你的key
   python3 -m venv .venv
   .venv/bin/pip install openai
   ./run_auto.sh --rules-file rules_zh.txt
   ```  
   若 Bridge 不在本机或端口不是 8081：  
   `./run_auto.sh --base-url http://Bridge机器IP:8081 --rules-file rules_zh.txt`

## 六、小结

- **实现链路**：Host（引擎 + Delegate）↔ Bridge（Player 实现 + HTTP API）↔ GPT Driver（OpenAI Function Calling + 三个 tools）。  
- **写接口**：在 Bridge 里加 Javalin 路由和 BridgeRuntime 的 getState/getLegalActions/applyAction；给 GPT 的「接口」就是 TOOLS 里的 function 定义 + system prompt。  
- **调用 GPT**：Driver 循环发 state/legal_actions 的 tool 结果 + 用户消息，模型返回 tool_calls，Driver 再发 POST /act，直到回合结束或断开。  
- **当前**：只完整支持**购买**和**部署**；**进攻、移动、运输、海战**需要扩展 Bridge 与 IMoveDelegate 的集成，并在 state/legal_actions/act 中增加移动相关字段与动作类型。  
- **部署**：保证 JDK、Python、OPENAI_API_KEY 和 Host/Bridge/Driver 的启动顺序与端口、`--take` 玩家名一致，即可在本地或跨机复现。
